# Процесс разработки Android-клиента «ЗОВ денег»

Документ фиксирует **порядок работ** над этим приложением: визуал по макетам из Figma (dev-плагин в монорепозитории) и сеть строго по **контракту REST** в [`openapi.yaml`](../../openapi.yaml). Детали стека — в [`android-stack.md`](android-stack.md).

## 2. Подготовка репозитория и окружения

1. Клонировать **монорепозиторий** (корень — один Git, без submodule).
2. Открыть в Android Studio **этот каталог** (`zov-android/`) как корень Gradle-проекта.
3. Убедиться, что версии JDK/AGP из проекта собираются локально (`assembleDebug`).
4. При правках макетов в Figma-плагине: из **корня** монорепы выполнить `node zov-figma/build.mjs`, затем пересобрать сцену в Figma по инструкции в `zov-figma/README.md`.

## 3. Архитектура и слои кода

Работаем в парадигме **чистой архитектуры** (подробности стека — в [`android-stack.md`](android-stack.md)):

1. **`ui` (Compose)** — только отображение и ввод; состояние экрана приходит из **ViewModel**; побочные эффекты (навигация, снэкбары) — по событиям из VM.
2. **`domain`** — **use case**-ы: один сценарий = один класс/функция с понятным именем; без Android- и без HTTP-типов в идеале.
3. **`data`** — **репозитории** агрегируют источники (сеть, кэш, моки); реализация HTTP — через **Ktor Client** по контракту OpenAPI.

Зависимости направлены **внутрь**: UI → domain ← data.

## 4. Пошаговый процесс по фичам и экранам

Для каждого экрана или пользовательского потока повторяем цикл:

### Шаг A. Сверка с макетом

1. Открыть соответствующий экран в Figma (страница «Экраны» после генерации плагином).
2. Зафиксировать **состояния**: загрузка, успех, пусто, ошибка, недостаточно прав — как в макете или по правилам дизайн-системы из «Компоненты».
3. Список **полей данных** и откуда они в API — только по `openapi.yaml` (имена полей, типы, единицы).

### Шаг B. Контракт данных и домен

1. Описать **use case**-ы (например: «войти», «получить портфель», «создать заявку»).
2. Определить **модели домена** (если отличаются от DTO) и маппинг в слое data.
3. Заложить **обработку ошибок** в терминах продукта (сообщения пользователю согласованы с UI-гайдом из макетов).

### Шаг C. Data-слой

1. Методы репозитория и вызовы Ktor **строго по путям и схемам** OpenAPI.
2. До появления реального бэкенда — **мок** на Ktor (MockEngine, фикстуры JSON или in-memory), чтобы UI и домен не зависели от сервера.
3. После появления API — переключение на реальный `baseUrl` через конфигурацию сборки (build type / flavor), без дублирования бизнес-логики.

### Шаг D. ViewModel и UI

1. ViewModel держит **UI state** (data class / sealed interface), принимает **намерения** пользователя (events/intents).
2. Верстка **Jetpack Compose** + **Material 3**: размеры, шрифты, цвета выравниваются с макетом; при необходимости донастраивается `Theme` приложения под токены из дизайна.
3. Проверка на **нескольких ширинах** в разумных пределах; эталон — **360 dp** по ширине макета.

### Шаг E. Завершение фичи

1. Ручная проверка сценария **по макету** и **по API** (или моку).
2. Прогон **Detekt**: `./gradlew detekt` из `zov-android/` (см. [`android-stack.md`](android-stack.md)).
3. Код-ревью: соответствие слоям, отсутствие утечек платформы в domain.

## 5. Версионирование и CI

- Повседневная разработка — в feature-ветках, merge в основную линию разработки по правилам команды.
- **Сборка на CI** при пуше в ветку **`release`** — см. раздел про CI в [`android-stack.md`](android-stack.md); детали workflow (артефакты, подпись release, кэш Gradle) уточняются при настройке пайплайна.
